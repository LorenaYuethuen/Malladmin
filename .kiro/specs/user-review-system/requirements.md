# 用户评论系统需求规范

## 1. 概述

为电商网站添加完整的用户评论系统，支持用户对商品进行评分、文字评论和图片上传，提升用户购物体验和商品信息透明度。

## 2. 用户故事

### US-1: 查看商品评论
**作为** 购物用户  
**当** 我浏览商品详情页时  
**系统应** 展示该商品的所有用户评论，包括评分、文字内容和图片  
**以便** 我可以了解其他用户的购买体验，帮助我做出购买决策

### US-2: 提交商品评论
**作为** 已购买商品的用户  
**当** 我收到商品后  
**系统应** 允许我提交评论，包括1-5星评分、文字描述和最多5张图片  
**以便** 我可以分享我的购物体验

### US-3: 编辑和删除评论
**作为** 已提交评论的用户  
**当** 我想修改或删除我的评论时  
**系统应** 允许我编辑评论内容或删除评论  
**以便** 我可以更正错误或更新我的评价

### US-4: 评论排序和筛选
**作为** 购物用户  
**当** 我查看商品评论时  
**系统应** 提供按时间、评分、有用性排序的选项  
**以便** 我可以快速找到最相关的评论信息

### US-5: 评论互动
**作为** 购物用户  
**当** 我阅读评论时  
**系统应** 允许我标记评论为"有用"或"无用"  
**以便** 帮助其他用户识别高质量评论

## 3. 验收标准

### AC-1: 评论列表展示
- **当** 用户访问商品详情页时
- **则** 系统应显示评论列表，包含：
  - 评论者用户名（脱敏处理）
  - 评分（1-5星，星级图标展示）
  - 评论文字内容
  - 评论图片（缩略图，点击可查看大图）
  - 评论时间
  - 有用/无用统计数
- **且** 默认按时间倒序排列
- **且** 每页显示10条评论，支持分页加载

### AC-2: 评论提交表单
- **当** 已登录用户点击"写评论"按钮时
- **则** 系统应显示评论表单，包含：
  - 评分选择器（1-5星，必填）
  - 文字输入框（10-500字，必填）
  - 图片上传区域（最多5张，每张最大5MB，支持JPG/PNG格式）
- **且** 提交前进行客户端验证
- **且** 提交成功后显示确认消息并刷新评论列表

### AC-3: 评分组件
- **当** 用户与评分组件交互时
- **则** 系统应：
  - 支持点击星星选择评分
  - 支持鼠标悬停预览评分
  - 显示当前选中的评分数值
  - 提供清晰的视觉反馈

### AC-4: 后端API - 评论CRUD
- **当** 前端调用评论API时
- **则** 系统应提供以下接口：
  - `GET /api/reviews?productId={id}` - 获取商品评论列表
  - `POST /api/reviews` - 创建新评论
  - `PUT /api/reviews/{id}` - 更新评论
  - `DELETE /api/reviews/{id}` - 删除评论
- **且** 所有接口返回标准JSON格式
- **且** 包含适当的HTTP状态码

### AC-5: 用户认证
- **当** 用户尝试提交、编辑或删除评论时
- **则** 系统应：
  - 验证用户登录状态
  - 验证用户是否购买过该商品（仅购买用户可评论）
  - 验证用户是否为评论作者（仅作者可编辑/删除）
- **且** 未授权操作返回401或403错误

### AC-6: 数据验证
- **当** 接收到评论数据时
- **则** 系统应验证：
  - 评分范围：1-5的整数
  - 文字长度：10-500字符
  - 图片格式：JPG或PNG
  - 图片大小：每张不超过5MB
  - 图片数量：不超过5张
- **且** 验证失败返回400错误和详细错误信息

### AC-7: 数据库设计
- **当** 系统存储评论数据时
- **则** 数据库应包含：
  - reviews表：id, product_id, user_id, rating, content, created_at, updated_at
  - review_images表：id, review_id, image_url, order
  - review_votes表：id, review_id, user_id, vote_type (useful/not_useful)
- **且** 建立适当的外键关系
- **且** 在product_id, user_id, created_at字段上建立索引

## 4. 约束条件

### 技术约束
- **若** 使用关系型数据库，**则** 必须支持事务处理
- **若** 上传图片，**则** 必须使用CDN存储，不存储在应用服务器
- **若** 评论数量超过1000条，**则** 必须实现分页和缓存机制

### 性能约束
- 评论列表加载时间不超过2秒
- 图片上传响应时间不超过5秒
- 支持并发100个用户同时提交评论

### 安全约束
- 所有用户输入必须进行XSS防护
- 图片上传必须进行病毒扫描
- 敏感词过滤：评论内容必须经过敏感词检测
- 用户个人信息（如用户名）必须脱敏显示

### 业务约束
- 用户只能对已购买的商品进行评论
- 每个用户对同一商品只能提交一次评论
- 评论提交后24小时内可编辑，之后只能删除
- 删除的评论保留30天后永久删除（软删除）

### 合规约束
- 必须符合数据保护法规（如GDPR、个人信息保护法）
- 用户有权删除自己的评论数据
- 系统必须记录评论的审计日志

## 5. 正确性属性

### P-1: 评分一致性
**属性**: 对于任何商品，其平均评分应等于所有评论评分的算术平均值（精确到小数点后1位）

**测试策略**: 
- 生成随机评论数据（1-100条）
- 计算期望平均分
- 验证系统返回的平均分与期望值一致

### P-2: 评论归属正确性
**属性**: 用户只能编辑或删除自己创建的评论

**测试策略**:
- 创建多个用户和评论
- 尝试用户A编辑用户B的评论
- 验证操作被拒绝（返回403错误）

### P-3: 购买验证正确性
**属性**: 只有购买过商品的用户才能对该商品进行评论

**测试策略**:
- 生成用户和商品数据
- 尝试未购买用户提交评论
- 验证操作被拒绝

### P-4: 数据完整性
**属性**: 删除评论时，相关的图片和投票数据也应被删除（级联删除）

**测试策略**:
- 创建包含图片和投票的评论
- 删除评论
- 验证review_images和review_votes表中相关记录也被删除

### P-5: 输入验证完整性
**属性**: 对于任何不符合验证规则的输入，系统必须拒绝并返回明确的错误信息

**测试策略**:
- 生成各种无效输入（评分超范围、文字过长/过短、无效图片格式等）
- 验证所有无效输入都被拒绝
- 验证错误信息准确描述问题

## 6. 非功能性需求

### 可用性
- 评论表单应提供实时字数统计
- 图片上传应显示上传进度
- 操作失败应提供友好的错误提示

### 可扩展性
- 系统架构应支持未来添加评论回复功能
- 应支持未来添加商家回复功能
- 应支持未来添加评论审核功能

### 可维护性
- API应提供完整的文档（OpenAPI/Swagger）
- 代码应包含单元测试（覆盖率>80%）
- 关键操作应记录日志

## 7. 依赖和假设

### 依赖
- 用户认证系统已存在
- 商品数据库已存在
- 订单系统已存在（用于验证购买记录）
- CDN服务可用（用于图片存储）

### 假设
- 用户已登录才能访问评论功能
- 商品ID在系统中唯一且有效
- 用户购买记录可通过订单系统查询
- 图片上传服务稳定可靠
